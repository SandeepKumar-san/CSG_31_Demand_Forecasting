ADAPTIVE TEMPORAL-STRUCTURAL FUSION MODEL FOR MINERAL COMMODITY DEMAND FORECASTING
________________________________________________________________________________

                              Chapter 6
                    Hardware, Software and Simulation

The practical implementation of the Adaptive Temporal-Structural Fusion (ATSF)
model for mineral commodity demand forecasting — covering the development
environment configuration, data preprocessing pipeline construction, model
training and multi-seed experimentation procedures, baseline comparison
protocol, risk scoring deployment, and Streamlit-based agentic web platform
development — is discussed in this chapter. All configuration parameters are
managed through a single experiments/config.yaml file, which is read at
runtime by src/utils/config.py. Zero hyperparameters are hard-coded in any
Python module, ensuring that every configuration change propagates
transparently from the YAML file to every component of the system without
requiring modification of production code.

6.1 DEVELOPMENT ENVIRONMENT
________________________________________________________________________________

The development environment was configured to support local debugging and
large-scale cloud-based training simultaneously, providing flexibility and
uninterrupted access throughout all phases of the project lifecycle.

6.1.1 PRIMARY HARDWARE CONFIGURATION:

A single NVIDIA T4 GPU hosted on Google Cloud Platform (GCP) Vertex AI
Workbench was the primary training infrastructure for all multi-seed
experiments. The complete hardware and software specifications, verified
verbatim from the reproducibility_report.json artifact generated after each
full training run, are as follows:

  • GPU Accelerator      : NVIDIA T4 (16GB GDDR6 VRAM)
  • CUDA Version         : 11.8 (matched to PyTorch cu118 build)
  • Computing Platform   : GCP Vertex AI Workbench
  • PyTorch Version      : 2.1.0+cu118 [22]
  • Graph Library        : PyTorch Geometric [23]
  • Programming Language : Python 3.10
  • NumPy Version        : 1.24.3
  • CUDA Available       : True (confirmed in reproducibility report)

Although all intensive multi-seed training runs were executed on the cloud
infrastructure, local workstations were used for code development, unit
testing, and preliminary debugging runs. The system_dry_run.py script was
executed locally as a 10-second fast-test prior to every overnight cloud
run, verifying that the complete pipeline initialises without out-of-memory
errors or tensor shape mismatches.

6.2 DATA PREPROCESSING IMPLEMENTATION
________________________________________________________________________________

The preprocessing pipeline is fully modularised into dataset-specific loaders,
each responsible for converting raw input files into normalised PyTorch tensors
suitable for heterogeneous graph-based temporal modelling. All preprocessing
scripts apply deterministic operations under the global seed set by the
SeedManager class before any data loading begins.

6.2.1 SUPPLYGRAPH PREPROCESSING:

The supplygraph_loader.py module loads five-feature daily CSV streams for 40
products covering 222 trading days from 2022 to 2023 [19]. The five feature
channels are: Sales Order, Production, Factory Issue, Delivery to Distributor,
and a derived operational_mask as the fifth feature channel. Per-node per-feature
Z-score normalisation is applied first, followed by clipping to the range
[-5, 5] to handle zero-inflated outliers that arise when products have no
sales activity on a given trading day. A strict 70/10/20 chronological split
is implemented using integer index slicing rather than sklearn's
train_test_split function, which is explicitly avoided to prevent data leakage
across time boundaries. The DataLoader is configured with num_workers=0
(single-threaded) to ensure compatibility with PyTorch Geometric graph objects
on GCP runtime environments, which do not support multiprocessing fork-based
data loading reliably.

6.2.2 USGS PREPROCESSING:

The usgs_preprocessing.py module parses the raw USGS Mineral Commodity
Summaries annual text reports [20] into structured tabular data, extracting
12 features per commodity per year. The 12 features, as defined in the
EDGE_TYPE_NAMES dictionary within usgs_loader.py (lines 44-60), are:
Production, Consumption, Imports, Exports, Price, Stocks, Supply, Employment,
WorldProduction, NetReliance, NetImportReliance, and RecyclingRate. The
usgs_loader.py module subsequently loads the cleaned annual data and applies
Z-score normalisation. To maximise the use of the extremely sparse five-year
temporal window (2019–2023), the loader implements a rolling-window cross-
validation strategy governed by the parameters rolling_window_cv=True and
n_cv_folds=2. This produces two training folds from the available annual
observations, ensuring that every node receives 3-4 training samples per fold
despite the low-data regime.

6.2.3 DATA INTEGRITY AND VERIFICATION:

Before any training begins, two dedicated QA scripts validate the integrity of
the prepared tensors. verify_data.py executes checks for NaN values, shape
mismatches between temporal tensors and graph edge indices, and normalisation
range violations for the SupplyGraph dataset. verify_usgs_integrity.py performs
equivalent checks for the USGS dataset, additionally validating that the 127-node
graph topology constructed by graph_builder.py contains all 15 expected typed
edge categories and that no node is disconnected from the main component. These
scripts terminate with a detailed error trace if any integrity check fails,
preventing silent data corruption from propagating into training runs.

6.3 MODEL IMPLEMENTATION
________________________________________________________________________________

The five-layer end-to-end ATSF pipeline is assembled in complete_model.py as
the AdaptiveFusionForecaster class. The forward pass executes six sequential
operations on each batch: (1) TFT Branch processes temporal sequences to produce
a temporal embedding and quantile predictions; (2) GAT Branch processes the full
supply-chain graph to produce structural node embeddings; (3) a node-to-batch
mapping slices the structural embedding by product_id to assemble per-instance
structural context; (4) the Adaptive Fusion Layer combines the two embeddings
via the learned scalar alpha; (5) the Fused Demand Representation maps the
fused embedding to point forecasts; and (6) the Risk Scoring Layer computes
procurement risk assessments when compute_risk=True, a flag that is disabled
during training to avoid unnecessary overhead. The model additionally supports
a forward_with_fixed_alpha(batch, fixed_alpha) method, which bypasses the Alpha
MLP entirely and forces a user-specified fusion ratio. This method generates all
three ablation baselines — TFT-only (alpha=1.0), GAT-only (alpha=0.0), and
Fixed Fusion (alpha=0.5) — from the same trained model weights without
re-training separate architectures.

6.3.1 TFT BRANCH (tft_branch.py):

The temporal branch implements the Temporal Fusion Transformer architecture [1].
Input features are first processed by the Variable Selection Network (VSN), which
applies soft learned feature gating to select the most relevant static and
dynamic covariates for each prediction step. The gated outputs are then fed
into a two-layer LSTM encoder with a hidden dimension of 128 for the SupplyGraph
dataset and 64 for the USGS dataset. Multi-head self-attention configured with
H=4 heads is applied over the LSTM hidden states to capture complex temporal
interactions beyond the range of the recurrent encoder. Three quantile heads
(P10, P50, P90) generate calibrated prediction intervals via a multi-quantile
pinball loss function [14][15]. The branch outputs a temporal embedding vector
of dimension H (the LSTM hidden dimension) alongside probabilistic quantile
forecasts for all configured horizons: {1, 3, 6, 12} days for SupplyGraph and
1 year for USGS.

6.3.2 GAT BRANCH (gat_branch.py):

The structural branch implements the EdgeTypeAwareGAT class, a two-layer
multi-relational Graph Attention Network [2][3][18]. Each node in the graph carries
four core features computed by graph_builder.py: (1) the recent demand average
(7-day rolling mean for SupplyGraph; annual mean for USGS), (2) a price-level
proxy (current price divided by historical average price), (3) production
capacity utilisation (recent production divided by peak production), and (4)
normalised lead time (lead_time_days / 30.0, clamped to [0, 1]).

Edge features are constructed by concatenating the one-hot encoding of the
edge type ID with a scalar edge confidence weight in [0.5, 1.0], yielding
feature tensors of dimension 5 for SupplyGraph (4 edge types + 1 weight) and
16 for USGS (15 edge types + 1 weight). The edge type embedding table is
initialised as an identity matrix (nn.init.eye_) to ensure maximum separability
at training onset and preserve the interpretability of individual relation types.

The two-layer architecture operates as follows: the first GATConv layer uses
head concatenation (concat=True) to produce an intermediate representation of
size hidden_dim × 4. The final GATConv layer averages across heads
(concat=False), producing the structural embedding of dimension hidden_dim.
Between layers, each GATConv output passes through BatchNorm1d (not LayerNorm,
as node-level batch statistics are more stable for graph data without a
sequential dimension), followed by ELU activation and dropout. The final layer
returns full edge attention weights, which are extracted for the downstream
edge-type importance analysis in Section 7.6. A lightweight node importance
scorer — a single nn.Linear(hidden_dim, 1) followed by Sigmoid — provides
per-node criticality scalars consumed by the risk scoring interface in Layer 5.

6.3.3 ADAPTIVE FUSION LAYER (fusion_layer.py):

The AdaptiveFusionLayer class implements the core architectural contribution
of this project: a context-conditioned scalar fusion weight alpha in (0, 1)
that is learned end-to-end with the rest of the model parameters. The fusion
proceeds in four steps. In Step 1, the temporal embedding h_t is passed through
a residual projection block — Linear(H, H) → LayerNorm → ReLU + h_t (residual
connection) — which aligns the temporal representation into a space more
comparable to the structural embedding while preserving the raw temporal signal.
In Step 2, two separate 16-dimensional embedding tables look up context vectors
for the material type (up to 8 categories) and forecast horizon (up to 4 options),
encoding domain-level prior knowledge about which modality is typically more
relevant for a given material class or planning horizon. In Step 3, the enriched
temporal embedding, structural embedding, and two context vectors are concatenated
into a single vector of dimension 2H + 32 and fed through a three-stage MLP:

    Layer 1: Linear(2H+32, H) → ReLU → Dropout(0.1)
    Layer 2: Linear(H, H/2)   → ReLU
    Layer 3: Linear(H/2, 1)

The raw scalar output is passed through a temperature-scaled sigmoid to compute
alpha, as given by Equation (1):

    Equation (1):  alpha = sigmoid(MLP([h'_t, h_s, z_m, z_h]) / tau)

where tau=2.0 for both datasets. The temperature greater than 1 flattens the
sigmoid curve, preventing premature saturation during early training when
gradients are large and noisy. In Step 4, the fused representation is computed
by Equation (2):

    Equation (2):  h-hat = alpha * h_t + (1 - alpha) * h_s

A post-fusion block (Linear(H, H) → LayerNorm) projects h-hat into the output
space before passing it to Layer 4. To prevent alpha from collapsing to a
degenerate extreme, a one-sided boundary regularisation term is added to the
training loss as given by Equation (3):

    Equation (3):  L_reg = lambda * (E[ReLU(alpha - 0.95)] + E[ReLU(0.05 - alpha)])

This double-hinge penalty applies only when alpha drifts outside the informative
[0.05, 0.95] range whilst freely permitting any value within that range,
including the strong unimodal preference alpha=0.919 observed on SupplyGraph.
Lambda is 0.1 for SupplyGraph and 0.05 for USGS, reflecting the richer structural
signal available in the annual USGS dataset and the correspondingly lower need
for boundary enforcement.

6.3.4 OUTPUT AND RISK LAYERS (fused_representation.py / risk_decision_layer.py):

Layer 4 — implemented as the Fused Demand Representation module — applies a
final Linear + LayerNorm projection to map the fused h-hat to point forecasts
across all configured horizons. The module also assembles the complete forecast
dictionary containing keys {forecast, std, lower, upper}, deriving the interval
bounds directly from the P10 and P90 quantile outputs of the TFT branch and
passing the full dictionary to Layer 5.

Layer 5 — implemented in risk_decision_layer.py as the RiskScoringLayer class —
translates point forecasts and GAT attention weights into procurement risk
assessments across three dimensions. Budget Stress is computed as
(forecasted_demand × price) / budget_threshold, where budget_threshold defaults
to 100,000 as set in config.yaml, and risk levels are assigned as: Low (<0.7),
Medium (<0.9), High (<1.1), and Critical (≥1.1). Lead-Time Risk is computed as
(forecast × lead_time_days / 30) / current_stock with a safety_factor of 1.5;
a ratio above 1.5 triggers the Critical level. Dependency Criticality uses the
mean incoming GAT attention weight per node, flagging nodes with a criticality
score above 0.7 as Critical. A lightweight learned calibrator MLP —
Linear(hidden_dim+3, 32) → ReLU → Linear(32, 1) → Sigmoid — combines the fused
embedding with the three raw risk scores to produce a single overall_risk_score
per product. It is explicitly disclosed that lead-time proxy values are
synthetic: SupplyGraph uses group-level constants sampled from [3, 30] days;
USGS uses a fixed value of 1.0. This layer is a proof-of-concept and has not
been validated against real procurement outcomes [Limitation 4].

6.4 TRAINING IMPLEMENTATION
________________________________________________________________________________

The trainer.py module implements the ReproducibleTrainer class. The combined
training objective is implemented in loss.py as the AdaptiveFusionLoss class:

    L_total = L_MSE + 0.3 * L_pinball + L_reg

where L_pinball is the multi-quantile pinball loss [14] for the P10, P50, and
P90 heads:

    L_pinball = mean(max(q * error, (q - 1) * error))  for q in {0.1, 0.5, 0.9}

and L_reg is the one-sided boundary regularisation from Equation (3). This
asymmetric pinball loss penalises underestimation by factor q and overestimation
by factor (1-q), incentivising the network to predict the true q-th percentile
of the demand distribution rather than simply minimising a symmetric error.

The Adam optimiser [21] is initialised with a learning rate of 1e-3. The
scheduler ReduceLROnPlateau reduces the learning rate by a factor of 0.3
whenever validation loss stalls for 10 consecutive epochs, with a minimum
learning rate floor of 1e-6 (SupplyGraph) and 1e-5 (USGS). Gradient clipping
is applied at L2 norm 1.0 after loss.backward() and before optimizer.step() on
every single batch, preventing LSTM recurrent gradient explosions during
Backpropagation Through Time on the 30-day SupplyGraph lookback window.

Deterministic seeding is managed by the SeedManager class in src/utils/seed.py,
which simultaneously locks all seven randomness sources before training begins:
Python random, NumPy, torch.manual_seed, torch.cuda.manual_seed_all,
cudnn.deterministic=True, cudnn.benchmark=False, PYTHONHASHSEED, and
CUBLAS_WORKSPACE_CONFIG. An additional per-epoch seed — torch.manual_seed
(seed_manager.seed + epoch) — is set at the start of each epoch to ensure that
dropout patterns and stochastic augmentation are reproducible without freezing
all stochastic diversity within an epoch. Per-batch alpha statistics (mean and
standard deviation) are tracked every epoch and logged to JSON files at
results/logs/training_history_seed{N}.json, providing full transparency into
alpha convergence dynamics. These logs are the primary data source for the alpha
trajectory plots of Figure 7.2 and the per-category alpha analysis of Table 7.1.
The training convergence statistics per seed are summarised in Table 6.2, read
verbatim from the training_history JSON log files.

6.5 BASELINE AND EVALUATION IMPLEMENTATION
________________________________________________________________________________

The baseline evaluation framework ensures strict comparability between classical
and neural approaches on identical held-out test sets. All scripts operate on
the same chronological 70/10/20 split (train/validation/test) used during ATSF
training, preventing any information leakage from future time steps.

run_classical_baselines.py implements ARIMA [24] and XGBoost [25] as
deterministic single-run baselines. Both are evaluated on the same held-out
test set — the final 20% of the chronologically ordered data — ensuring a
directly comparable evaluation horizon.

run_graph_baselines.py uses the forward_with_fixed_alpha method of
AdaptiveFusionForecaster to generate TFT-only (alpha=1.0), GAT-only (alpha=0.0),
and Fixed Fusion (alpha=0.5) results from the identical trained model weights.
This design guarantees ablation consistency: all three ablation variants share
~198,000 trainable parameters with the full ATSF model, so any performance
difference is attributable entirely to the choice of fusion strategy rather than
to differences in model capacity.

evaluate.py loads each seed's best checkpoint and evaluates on the held-out
test set with model.eval() and torch.no_grad() to disable dropout and prevent
gradient graph construction, which would cause GPU out-of-memory errors during
evaluation on large batches. The script assembles P10-P90 coverage statistics
from quantile outputs and writes the complete per-seed evaluation report,
including WAPE, RMSE, SMAPE, R2, and P10-P90 coverage, to the results directory.

edge_type_analysis.py iterates over all 15 USGS edge types. For each type, it
constructs a modified graph excluding all edges of that type and re-evaluates
the trained ATSF model on the test set, measuring the change in RMSE relative
to the full-graph baseline (RMSE=0.6317). Negative Delta RMSE indicates that
removing the edge type degrades performance — the edge is causally important.
Positive Delta RMSE indicates that removing the edge type improves performance —
the edge adds noise at annual temporal resolution.

stats_analysis.py computes the one-tailed Wilcoxon signed-rank test [26] across
the five seed pairs for ATSF versus each baseline, and calculates the 95%
confidence interval for USGS WAPE via bootstrap resampling of the five seed
measurements.

6.6 RISK SCORING AND VISUALISATION
________________________________________________________________________________

demo_risk_scoring.py runs the complete ATSF forward pass with compute_risk=True,
triggering the full Layer 5 computation for all test-set nodes. For each product
node, the script outputs a JSON dictionary containing: product_id, forecast,
forecast_std, budget_stress, budget_risk, leadtime_risk_score, leadtime_risk,
dependency_criticality, dependency_risk, overall_risk_score, and the list of
action flags. Four action flags are generated by rule-based thresholding:
"[WARN] Review budget allocation" (if budget_risk is High or Critical);
"[URGENT] Expedite supplier" (if leadtime_risk is Critical); "[WARN] Consider
increasing order quantity" (if leadtime_risk is High); and "[CRITICAL] Activate
backup supplier" (if dependency_risk is Critical). If no flags trigger, the
output is "[OK] All risk levels acceptable."

The visualisation pipeline consists of two dedicated scripts. gen_alpha_fig.py
generates the two-panel alpha trajectory plots (Figure 7.2) by reading the
per-epoch alpha_mean values from the training_history JSON logs across all five
seeds for both datasets. gen_edge_attention_fig.py extracts the edge attention
weights from the trained GAT branch for the USGS dataset, aggregates mean
attention per edge type across all test-set edges, and generates Figure 7.1.
Both scripts produce publication-grade figures using Matplotlib with configured
DPI and tight layout settings to meet IEEE figure quality standards.

6.7 INTERACTIVE WEB PLATFORM AND AGENTIC PIPELINE
________________________________________________________________________________

To bridge the usability gap between raw graph neural network tensor outputs and
non-technical procurement stakeholders, the ATSF project delivers a fully
agentic web platform built on the Streamlit framework and hosted at
localhost:8501 during development. The platform implements a
Retrieval-Augmented Generation (RAG) architecture that fuses the deterministic
empirical outputs of the ATSF model — structural ablation metrics, edge
attention scores, per-seed alpha values — with a large language model to
translate complex topological information into plain-language strategic
procurement intelligence. The dashboard is organised into four functionally
distinct tabs.

6.7.1 DYNAMIC DATA AGGREGATION (Tabs 1 and 2):

The first two tabs implement autonomous cross-seed data aggregation. On
startup, the system loads multi_seed_metrics.csv and classical_baselines.json
from both the SupplyGraph and USGS results directories. Using Plotly Express
and Plotly Graph Objects, the dashboard renders interactive performance
distribution violin plots for all five seeds across all six models, directly
visualising the key finding of the project: the mean alpha convergence gap of
0.919 (SupplyGraph) versus 0.489 (USGS). Edge-ablation impact charts
differentiate causally important structural edges such as technology_cluster
(Delta RMSE = -2.20%) and substitution (Delta RMSE = -0.39%) from systemic
noise edges such as supply_chain_input (Delta RMSE = +1.29%) and
recycling_secondary_source (Delta RMSE = +0.50%). All charts are fully
interactive, allowing drill-down by seed, by dataset, and by baseline model at
runtime.

6.7.2 DUAL-LLM PROCUREMENT AGENT (Tab 3):

The third tab implements a three-step real-time procurement intelligence
pipeline powered by the meta/llama-3.1-70b-instruct model accessed via the
NVIDIA NIM API at a deterministic temperature of 0.2, which minimises response
stochasticity to ensure businessconsistent outputs across identical user queries.
When a user submits a natural-language business objective through the Streamlit
text area (for example, "we are building a lithium-ion battery manufacturing
facility"), the system executes the following orchestration steps.

In Step 1 — Entity Extraction — a structured system prompt forces the LLM to
parse the submitted scenario into a comma-separated list of required mineral
commodities rather than producing free-text advice. This structured output is
then programmatically split and each extracted commodity name is cross-referenced
against the 127 nodes in the native NodesIndex.csv lookup table. Exact string
matching is attempted first; if no exact match is found, fuzzy matching with a
Levenshtein distance threshold is applied to handle minor spelling variations.
Only commodity names confirmed to exist in the ATSF node index are retained,
preventing hallucinated commodity names from propagating into the downstream
graph query.

In Step 2 — External Concept Graph — the LLM is prompted with a "Supply Chain
Strategist" persona to generate broad strategic procurement advice for the
confirmed commodity list, including geopolitical risk narrative, substitution
considerations, and dual-sourcing recommendations. Simultaneously, the model is
tasked with generating a JSON-structured concept graph representing generic
domain knowledge relationships between the commodities mentioned, to be rendered
in the UI as an explanatory structural overlay.

In Step 3 — ATSF Native Subgraph Extraction — to prevent hallucination and
ground the system's output in the verified empirical data of the ATSF model,
a secondary deterministic pipeline extracts a topological trace directly from
the empirical all_edges.csv adjacency matrix. The system isolates all edges
involving the confirmed extracted commodity nodes, applies a confidence
threshold of edge_weight ≥ 0.8 to retain only high-confidence structural
relationships, and caps the result at the 35 strongest links to maintain UI
rendering stability. This extracted native subgraph contains only edges
that were present in the training data and evaluated by the GAT branch,
ensuring that no fabricated structural dependencies are presented to the user.

6.7.3 AGENTIC MATHEMATICAL TRANSLATION:

The raw ATSF structural trace generated in Step 3 is injected directly into the
context of a final LLM call that uses a "Data Science Lead" persona with
explicit instructions to explain precisely why the specific graph phenomena
observed in the extracted subgraph — such as high-attention edge types
identified by ablation analysis, or the specific alpha value associated with
the queried commodity — trigger supply chain volatility for the user's stated
business objective. This forced coupling between the LLM's narrative output
and the verified mathematical results of the ATSF model guarantees that
end-users receive procurement risk summaries that are grounded in the empirical
structural evidence of the project. The system explicitly prohibits the LLM
from introducing supply chain facts not present in the injected ATSF context,
thereby implementing model-grounded generation rather than unconstrained
internet-knowledge retrieval.

6.7.4 PHYSICS-BASED GRAPH ENGINE (Tab 4):

To enable interactive exploration of the full USGS supply-chain network, which
contains 6,000+ edges across 127 nodes and 15 typed edge categories, the
dashboard replaces static SVG diagrams with a real-time force-directed physics
simulation implemented using PyVis (vis.js backend). Users can filter the
displayed network by any combination of the 15 exact ATSF edge type hierarchies
established in Section 5.5 and Table 7.5, enabling targeted exploration of
specific structural relationships such as alloy_components or
geopolitical_supply_risk. The physics engine is configured with parameters
that autonomously arrange complex multi-node clusters: node_distance=180,
central_gravity=0.1, spring_length=150, and damping=0.09. These values were
selected to prevent visual overlap in dense regions of the commodity network
while maintaining sufficient spring tension to cluster structurally related
nodes. The simulation supports interactive drag-and-drop node repositioning,
and each node displays its commodity name and trained GAT attention score on
hover, enabling non-technical users to directly inspect the structural evidence
underlying the ATSF model's procurement risk assessments.
